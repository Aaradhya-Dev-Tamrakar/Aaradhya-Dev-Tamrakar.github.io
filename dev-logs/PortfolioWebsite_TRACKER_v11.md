### What Was Decided / Learned
*Additive to v6's two lessons.*
- **The "verify before trusting a dead session" lesson from v6 held up again, and caught a real bug this time.** `c06c229e` didn't just verify its own merge — on the user's explicit prompt, it re-audited independently of its own prior turn's summary and found a genuine miss (achievements.html's light-theme bug) that had survived one full merge-and-verify cycle because the first pass worked off a *named-item list* inherited from a dead session's thinking trace rather than an exhaustive sweep. **Standing lesson, sharper now**: verifying "did my merge do what I intended" is not the same check as "does this file actually have every fix it should" — the first can pass cleanly while the second still finds gaps, specifically when working from someone else's (even a prior Claude turn's) list of what to check rather than an independent sweep.
- **Two sessions can each build a legitimate, complete, confirmed-delivered artifact from the same starting point without either being aware the other exists — and one can become an input to the other without documentation of that fact.** `6953f6fd` and `235b2f5c` are the clearest example yet: both fully delivered, both correct in isolation, and only reconcilable after the fact by close textual comparison. This is the same failure mode v6 named, but v6 saw it as *wasted duplicate effort*; here the duplicate effort was **complementary rather than redundant** (one did nav+webp+CSS-architecture, the other did color+readability+heading) and got combined into something more complete than either alone — but only because a later session happened to be handed both zips together. Nothing structural made that happen; it was luck of what the user uploaded together. Don't rely on it recurring.
- **Chain of custody for zips matters and usually isn't stated.** No file in this project has ever said "here is session X's output" in so many words. Every provenance link in this tracker (v6's and this version's) has been reconstructed from internal evidence — matching cutoff phrases, matching described architecture, matching a decision's exact wording. This works, but it's forensics, not a system. If the person wants tracking to stay reliable, naming zips by their producing session (not just by topic) would remove the need for it. **User-supplied addendum (post-v7, verbal, not from an export)**: the mechanical root cause is that length limits forced individual threads into serialized "Continue" chains while the user was also running multiple sessions in parallel — both effects compound the same way, and neither is visible from inside any single session. This is exactly why `6953f6fd`/`235b2f5c` and, now, `c06c229e`/`a59f8fd1` could each build something complete without knowing the other existed.
- **A narrated dead session is not evidence of anything, and this project has now hit that exact failure three separate times.** v6 had one (a session that produced nothing after a final "Continue"). v7's `3d6edf7a` had one (empty response, no output file, the next session had to rebuild from scratch). v8's `a59f8fd1` is a third: a full first-person narration of sed commands, meta-tag fixes, and a validation summary — indistinguishable in tone and detail from a session that actually did the work — with the export cutting off mid-tool-call and zero bytes ever written anywhere. **The fix that's worked every time so far is the same one**: don't trust a session's own account of its output, check the actual file. `c06c229e` happened to be doing the identical fix in parallel and actually finished it, which is the only reason v8 has real content to add here — if only `a59f8fd1` had been uploaded, this version would have nothing to show for the domain-identity item except another correctly-skeptical "no" in the State of Play table.
- **Asking beats guessing, again.** `c06c229e` found the repo internally consistent but consistently *wrong* (self-identifying as Repo A everywhere) and correctly treated "flip it unilaterally" as a worse move than "ask, since the tracker already flagged this as undecided" — a repo that's wrong-but-consistent is safer to sit with for one more turn than a repo that's inconsistently half-fixed. Same instinct `6953f6fd` used for the CSS-consolidation scope question in v7; holding now as a repeated pattern, not a one-off.
- **The forensic-reconstruction method this tracker has leaned on since v6 just got its first real external check, and it held up.** Every "confirmed delivered" claim from v7 and v8 was inference from session summaries and diffs against source zips — never the live repo. v9 finally had the live repo, checked the specific claims directly, and every single one came back true, byte-for-byte, including exact counts (8/20/15 heading swaps) that would have been easy to get subtly wrong if the underlying method were sloppy. This doesn't retroactively excuse skipping verification in future versions — it's one snapshot, not a standing guarantee — but it's real evidence the discipline this tracker has enforced (verify against files, not summaries; don't trust dead sessions; ask instead of guessing) has been producing an accurate picture, not just a self-consistent one.
- **Catching your own false alarm before reporting it is now a repeated, not isolated, pattern worth naming on its own.** `c06c229e`'s CRLF check in v145 and this version's canvas-color grep both did the same thing: a check came back looking like a regression, and rather than reporting "found a bug," the next step was re-verifying against a known-good baseline first — which in both cases dissolved the alarm (stale assumption; wrong regex pattern) rather than confirming it. Distinct from the "verify before trusting a dead session" lesson — this one is about not trusting your own first negative result either, positive or negative.
